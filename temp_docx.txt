Multi-Tenant Multi-Role Architecture for EdgeOps SaaS Platform
Overview
The EdgeOps platform follows a hierarchical multi-tenant architecture designed for organizations that manage multiple brands and locations. The system must support:
Multiple Organizations per User
Multiple Brands per Organization
Multiple Locations per Brand
Role-Based Access Control (RBAC)
Organization Switching
Brand Switching
Location Switching
Fine-Grained Permissions
Secure Multi-Tenant Data Isolation

1. Business Hierarchy
Platform Structure
Platform│├── Organization A│   ││   ├── Brand 1│   │   ├── Location 1│   │   ├── Location 2│   │   └── Location 3│   ││   └── Brand 2│       ├── Location 4│       └── Location 5│└── Organization B    │    ├── Brand 3    └── Brand 4
Hierarchy Rules
Organization is the tenant boundary.
Brands belong to a single Organization.
Locations belong to a single Brand.
Users may belong to multiple Organizations.
Users may have different roles in different Organizations.

Users may belong to multiple organizations and therefore require membership-based access.

10. User Session Context
After login, build a complete access tree.
Example:
{  "user_id": "123",  "organizations": [    {      "id": "org1",      "role": "owner",      "brands": [        {          "id": "brand1",          "role": "admin",          "locations": [            {              "id": "loc1",              "role": "manager"            }          ]        }      ]    }  ]}
This structure should be cached in the frontend session.

11. Context Switching
The user interface should always maintain:
Current OrganizationCurrent BrandCurrent Location
Example:
Organization: Restaurant Group ABrand: Pizza HouseLocation: New York
All API requests must respect the currently selected context.

12. Permission-Based Security
Avoid direct role checks.
Bad Example:
if(role === "owner")
Recommended:
if(hasPermission("users.manage"))

permissions
id UUID PRIMARY KEYkey TEXT UNIQUEdescription TEXT
Examples:
organization.managebrand.managelocation.manageusers.managereports.viewreports.exportbilling.manageinvoice.viewinvoice.editstaff.managesettings.manage

role_permissions
role TEXTpermission_key TEXT
Benefits
Easier maintenance
Future scalability
Enterprise flexibility
Custom role support

14. RLS Strategy
Access should be granted only when the authenticated user belongs to the corresponding hierarchy level.
Example:
User belongs to OrganizationANDUser belongs to BrandANDUser belongs to Location
Only then should records be visible.

Final Architecture Summary
Platform│├── Organizations (Tenant)│├── Brands│├── Locations│├── Users│├── Organization Memberships│├── Brand Memberships│├── Location Memberships│├── Roles│├── Permissions│└── RLS Policies
