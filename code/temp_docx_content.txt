PWA Service Worker Fix Plan (Critical Launch Blocker)
Problem Summary
The platform is experiencing a caching issue where:
Users opening the application in Incognito mode receive the latest deployment.
Users opening the application in a normal browser session receive an older version of the platform.
A hard refresh (Ctrl + Shift + R / Ctrl + F5) immediately loads the latest version.
This confirms that the issue is caused by Service Worker and browser caching, not by Vercel deployment or DNS configuration.

5. Cached index.html Causes Permanent Stale Versions

Users keep receiving old HTML.

New deployments may never be loaded until cache is manually cleared.
This is the primary reason users see different versions of the platform.

Recommended Long-Term Solution
Replace Custom Service Worker with vite-plugin-pwa
Current architecture uses a manually maintained Service Worker.
Recommended migration:

 install vite-plugin-pwa
Benefits:
Automatic Workbox integration.
Automatic Vite asset hashing support.
Automatic cache invalidation.
Automatic Service Worker lifecycle management.
Built-in skipWaiting and clients.claim support.
Safer deployment process.
Reduced maintenance burden.
Priority
Priority Level: P1 (Must Fix Before Public Launch)
Reason:
This issue directly affects deployment reliability and can result in customers using different versions of the platform simultaneously.
The platform should not be publicly launched until Service Worker update handling and cache invalidation are corrected.
Acceptance Criteria
After implementation:
✅ Users receive the latest deployment automatically.
✅ Normal browser and Incognito show the same version.
✅ No manual cache clearing required.
✅ No Ctrl + F5 required.
✅ Old caches are removed automatically.
✅ Service Worker updates activate immediately.
✅ Deployments become visible to users without intervention.
